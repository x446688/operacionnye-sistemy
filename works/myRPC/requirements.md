# Общие требования для проекта myRPC
1. В проекте с домашними заданиями создайте новый подпроект myRPC. Настройте файл «.gitignore», чтобы из вашего локального репозитория не попадали лишние файлы. 

2. Для ведения проекта должен применяться подход Git Flow (работы, в которых каждая новая фича не будет реализована в отдельных ветках не будут приниматься к рассмотрению).

3. Описание в формате Markdown.

4. Все реализованные программы в данном задании должны быть разработаны с учетом данных требований: 
https://www.gnu.org/prep/standards/html_node/Writing-C.html

## Используемый протокол (JSON протокол)
- Формат запроса на выполнение команды:
```json
{“login”:”имя_пользователя”,”command”:”команда bash”}
```
- Формат ответа сервера:
```json
{“code”:код_ответ, ”result”:”результат исполнения или описание ошибки”}
```
Код_ответа – 0, в случае успеха, в поле result записываются данные stdout
Код_ответа – 1, в случае ошибки, в поле result записываются данные stderr

## Общая схема работы

### Клиент myRPC-client  
Пользователь при помощи аргументов командной строки в утилите `myRCP-client ` задает параметры подключения к серверу, тип сокета и команду на языке BASH.
Утилита получает имя пользователя, от имени которого она запущена, подключается через механизм сокетов к серверу,
если она смогла успешно подключиться к серверу передает ему в выбранном формате JSON имя пользователя и команду.
### Сервер myRPC-server
Во время запуска получает из конфигурационного файла `/etc/myRPC/myRPC.conf` номер порта и тип сокета (потоковый или датаграмный)
получает через механизм сокетов строку в выбранном формате с именем пользователя и командой
разбирает строку, получает из нее имя пользователя и команду
проверяет в конфигурационном файле `/etc/myRPC/users.conf`, есть ли пользователь в списке допущенных пользователей
если пользователь есть в списке выполняет команду, записывая весь вывод stdout во временный файл по шаблону `/tmp/myRPC_ХХХХХХ.stdout`, а весь вывод ошибок во временный файл по шаблону `/tmp/myRPC_ХХХХХХ.stderr`. 
## Технические требования
Данный механизм должен использовать `syslog` (в данном примере - `mysyslog`) или заданный в аргументах командной строки файл для вывода ошибок в журнал.  
`myRCP-client` должен быть консольный приложением на языке Си и принимать заданные выше аргументы командной строки;  
`myRCP-server` в зависимости от конфигурации запускается как консольное приложение или как программа-демон. Программа должна обрабатывать сигналы от Ctrl+C, сигнал SIGHUP для обновления конфигурации и сигналы от дочерних процессов, о факте получения сиггналов писать в журнал;  
Перечень допущенных пользователей задается их перечислением в файле `/etc/myRPC/users.conf`, где на каждой строке задается имя одного пользователя
При передаче специальные символы (например, двойные кавычки) в строке с командной должны экранироваться.   
### Аргументы командной строки `myRCP-client`

__Команда BASH__
`-с` ”команда bash” или `--command` ”команда bash”  

__Адреса сервера__
`-h` ”ip_addr” или `--host` ”ip_addr”

__Порт сервера__
`-p` 1234 или `--port` 1234

__Тип сокета__  
`-s` или `--stream` – использовать потоковый сокет  
`-d` или `--dgram` – использовать датаграмный сокет  

__Вывод помощи__  
`--help`

### Пример содержимого конфигурационного файла
```conf
#comment  
port = 1234  
#socket_type = dgram  
socket_type = stream  
```

### Пример содержимого файла с пользователями
```conf
user  
tester  
developer  
dolphin
```

## Правила сборки и дистрибуции

1. Все программы собираются при помощи Makefile со следующими секциями:

`all` – собирает все цели;  
`clean` – удаляет все временные и объекты файлы, создаваемые при компиляции, исходные тексты должны приводится к изначальному виду;  
`deb` – собирает deb-пакет для программы.  

2. Каждому модулю (программе и библиотеки) внутри подпроекта сделан отдельный файл readme.md на языке Markdown с описанием правил сборки и использования данного модуля.

3.  Создается общий Makefile, который собирает все программы.

4.  Добавлены в репозиторий deb-пакетов из [задания №2](../mysyslog/README.md) deb-пакеты myRCP-client и myRCP-server.

5.  Должен быть обновлен кэш пакетов на клиенте и сервере.

5.  Установлены пакет myRCP-client на клиенте и myRCP-server на сервере.

6.  Добавлены пользователи в список разрешенных пользователей на сервере и настроен конфигурационный файл. Запущен myRCP-server как служба.

7.  Отправлены различные команды с клиента на сервер, как от разрешенных пользователей, так и от – запрещенных.
